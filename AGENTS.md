# 项目协作规范

本目录用于协作完成《SaaS 类 AI 工具赋能基础办公》课程。任何新 task、agent 或人工协作者进入目录后，必须遵循本文件。

## 开始工作前

按顺序完整阅读：

1. `项目说明.md`
2. `00-项目管理/项目总览与当前进展.md`
3. `01-内容策划/原始想法.md`
4. `01-内容策划/PPT大纲.md`
5. `01-内容策划/页面台账.md`
6. 与当前任务直接相关的研究资料和最近一条 `00-项目管理/任务交接记录.md`

不要要求用户重新讲述这些文档中已经记录的背景、决定和进度。

## 内容边界

- 课程面向跨部门业务骨干，约 60 分钟。
- 主线是“理解 Agent → 认识 WorkBuddy → 看懂案例 → 学会迁移方法”。
- 第一版主课件不使用出版行业专属案例；出版案例只能进入备选案例库。
- 主案例通过预先录制的视频演示，不安排学员现场同步实操。
- 任何案例录屏前，必须先说明场景、输入、过程、产物、人机分工和验收标准。
- WorkBuddy 是主要案例工具；Codex 只作为工具版图、能力对照或备选案例出现。
- 未经官方资料或录屏环境实测确认，不得把具体 Skill、插件、界面名称写成确定事实。

## 文件职责

- 原始想法只追加或标注状态，不删除用户原意。
- 已确认结构写入 `PPT大纲.md`；未确认内容写入 `原始想法.md` 或 `备选案例库.md`。
- 每次形成实质决定，更新 `00-项目管理/决策记录.md`。
- 每次新增资料、链接、截图或附件，更新 `02-资料研究/资料索引.md`。
- 每次完成一个 task，更新项目进展、页面台账和任务交接记录。
- `项目总览与当前进展.md` 是项目状态的唯一真源，不在多个文件维护互相冲突的“当前状态”。

## 版本和产物

- HTML 源代码只维护在 `04-网页版PPT/课件源代码/`，不得按 v1、v2 复制整套目录。
- 讲稿只维护当前源文件 `05-讲稿/讲稿.md`；它与课件代码在同一个 Git commit/tag 下对应。
- 日常版本由 Git commit 管理；阶段版本使用 `ppt-YYYY-MM-DD-vN` tag 冻结。
- 只有 PPTX、PDF、离线 HTML 包等发布物使用 `YYYY-MM-DD-vN-SaaS类AI工具赋能基础办公` 文件名。
- 每次发布必须在 `06-交付成果/发布记录.md` 登记 tag、commit 和各交付物对应关系。
- 讲稿逐页记录：讲述内容、建议时间、录屏时机、观看重点、可跳过内容、互动备用话术。
- 网页课件使用 16:9 冷白调研风；素材优先 3840×2160，最低 1920×1080。
- 图示保留 `.drawio` 源文件；录屏同时保留关键帧截图。

## 来源与安全

- 原始资料不可直接改写或覆盖；派生内容另存。
- WorkBuddy 白皮书标记为内部参考，未经确认不直接把其整页或敏感信息放入公开课件。
- 涉及产品版本、价格、下载方式、支持平台和功能状态时，优先核对官方最新资料。
- 案例只使用脱敏、模拟或明确授权的数据。
- 事实、数字、人名、日期和产品能力必须可追溯；宣传性表述不得直接当作已验证事实。

## Git 约定

- Markdown、HTML、JSON、脚本和 draw.io 源文件正常跟踪。
- PDF、DOCX、PPTX、视频等二进制大文件使用 Git LFS。
- 忽略 Office 锁文件、缓存、临时渲染和本机预览状态。
- 不提交密码、令牌、Cookie、个人数据或录屏账号信息。
